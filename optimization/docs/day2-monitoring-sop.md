# Day 2 监控补齐 SOP（可观测性建设）

适用范围：生产服务器接管第 2 天。
目标：把“能看见、能告警、能定位”建立起来。

## 1. Day 2 验收目标（DoD）
- 有统一指标入口（Prometheus 或等价监控平台）。
- 有主机 + 应用 + 数据库/缓存三层核心指标。
- 有 P1/P2 告警规则，且已打通通知通道。
- 有值班 Runbook 链接（告警触发后可执行）。

## 2. 指标分层标准（必须覆盖）

### 2.1 主机层（Host）
- CPU：usage、load、steal、iowait
- 内存：used、available、swap、major fault
- 磁盘：usage、inode、iops、await、util
- 网络：bandwidth、retransmit、drop、tcp 状态分布

### 2.2 应用层（Service）
- 流量：QPS / RPS
- 延迟：P50/P95/P99
- 错误：4xx/5xx、异常率
- 饱和：线程池、连接池、队列积压

### 2.3 依赖层（DB/Cache/MQ）
- MySQL：连接数、慢查询、锁等待、复制延迟
- Redis：命中率、内存、慢命令、bigkey/hotkey
- MQ：积压长度、消费延迟、失败率

## 3. 告警分级建议（首版）

### P1（立即响应）
- 可用性下降（5xx 突增 / 探活失败）
- 延迟严重超 SLO（持续 5 分钟）
- 关键实例宕机
- 磁盘 > 90% 或 inode > 85%
- 发生 OOM / NTP 失步

### P2（尽快处理）
- CPU 高位持续
- iowait 偏高
- 连接池接近上限
- 慢查询持续升高

### P3（观察优化）
- 单项资源趋势异常但未影响 SLA

## 4. 监控接入顺序（推荐）
1. 主机指标（node_exporter）
2. 应用指标（/metrics 或 APM agent）
3. DB/缓存 exporter
4. 黑盒探测（HTTP/TCP）
5. 告警与值班流程联动

## 5. Dashboard 最小看板（必须有）
- 全局总览：可用性、QPS、P95、错误率
- 主机看板：CPU/内存/磁盘/网络
- 服务看板：实例数、流量、延迟、错误、饱和
- 依赖看板：MySQL/Redis 关键指标

## 6. 值班联动
- 每条 P1 告警必须绑定：负责人、SOP 链接、止血步骤。
- 告警内容要有上下文：实例、环境、近 15 分钟趋势、关联变更。

## 7. Day 2 产出物
- `监控资产清单`（采集目标、标签、负责人）
- `告警规则清单`（阈值、持续时间、级别）
- `值班 Runbook`（定位与止血路径）
- `可观测性覆盖率`（服务接入率、告警可行动率）

## 8. 执行原则
1. 先覆盖关键路径，再追求全面。
2. 告警宁少而准，避免告警风暴。
3. 没有负责人和动作的告警，等于无效告警。

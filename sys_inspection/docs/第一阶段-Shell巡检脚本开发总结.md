# 第一阶段（Shell）巡检脚本开发与维护手册

> 文档定位：第一阶段脚本的**实现说明 + 运维手册 + 变更记录**  
> 适用对象：脚本维护者、运维值班同学、巡检平台对接人员

---

## 1. 背景与目标

第一阶段的目标是：在不安装 Agent 的前提下，通过一台控制机批量巡检 Linux 服务器，输出可追溯、可归档、可比对的报告。

核心要求：
- 可批量并发执行
- 结果可读且结构化
- 异常可定位（日志 + 原始输出）
- 兼容不同 Linux 发行版与不同工具安装情况

---

## 2. 脚本入口与目录职责

## 2.1 入口文件

- `inspect.sh`：巡检主流程入口，负责参数解析、依赖检查、目标选择、并发执行、汇总统计、报告输出。

## 2.2 配置目录

- `config/servers.csv`：目标主机清单（可含账号密码与业务信息）
- `config/threshold.conf`：阈值配置

## 2.3 模块目录

- `lib/check_deps.sh`：依赖检查与交互式处理
- `lib/system_check.sh`：系统基础信息
- `lib/cpu_check.sh`：CPU/负载
- `lib/memory_check.sh`：内存/Swap
- `lib/disk_check.sh`：磁盘
- `lib/network_check.sh`：网络连接
- `lib/process_check.sh`：进程与服务
- `lib/security_check.sh`：安全相关（登录失败、防火墙、SELinux）

---

## 3. 主流程（运行时序）

脚本主流程如下：

1. 参数解析（`-a/-m/-r/--host/...`）
2. 打印 Banner
3. 获取巡检人（`--inspector` 或交互输入，默认 `whoami`）
4. 依赖检查（`check_deps.sh`，交互处理缺失工具）
5. 加载目标主机（`--host` 单机 or `servers.csv` 批量）
6. 并发巡检（本地/远程）
7. 结果回收与状态归一化
8. 汇总统计（OK/WARNING/CRITICAL）
9. 生成报告（html/text/json）
10. 可选邮件发送

---

## 4. 参数体系（维护视角）

关键参数说明：

- 批量巡检：`-a, --all`
- 单机巡检：`--host --port --user --password`
- 报告类型：`-r html|text|json`
- 模块选择：`-m cpu,mem,disk,net,proc,sec,sys`
- 并发控制：`-p`
- 超时控制：`-t`
- 巡检人：`--inspector`

维护建议：
- 自动化场景（CI/Cron）建议显式传 `--inspector`
- 生产环境建议显式传 `-t` 与 `-p`，避免默认值不适配

---

## 5. 依赖检查机制（交互菜单）

当依赖缺失时，脚本提供统一交互：

- `[1]` 显示安装命令（手动安装后继续）
- `[2]` 使用替代方案继续（功能可能受限）
- `[3]` 显示详细说明（安装命令 + 替代方案）
- `[4]` 忽略并继续（跳过依赖相关功能）
- `[0]` 退出脚本

该机制保证：
- 有严格模式（尽量安装）
- 有降级模式（替代方案）
- 有应急模式（忽略）

## 6. 近期关键修复说明（重点）

以下修复已落地，作为后续维护基线：

## 6.1 `--host` 单机目标加载修复

问题：单机参数在某些路径下会被后续逻辑覆盖或清空。  
修复：单机目标与批量目标分支彻底分离，确保单机配置不会被覆盖。

## 6.2 状态聚合与展示一致性修复

问题：状态字段异常时，汇总与报告展示不一致。  
修复：新增状态归一化处理，未知状态不再误判为正常，统一按 `WARNING` 处理并记录异常说明。

## 6.3 `LOGIN_FAIL_IPS` 解析与展示修复

问题：安全模块的登录失败 IP 统计在部分链路未被解析/渲染。  
修复：纳入白名单解析，并在报告中稳定展示。

## 6.4 SSH 兼容性修复（`accept-new`）

问题：老版本 OpenSSH 不支持 `StrictHostKeyChecking=accept-new`。  
修复：运行时探测 SSH 选项兼容性，不支持时自动降级为 `StrictHostKeyChecking=no`。

## 6.5 `pipefail` 下计数异常修复

问题：`grep -c ... || echo 0` 在 `pipefail` 场景可能产生 `0\n0`，影响数值处理。  
修复：改为更安全写法（如 `|| true` + 兜底处理），避免脏值。

## 6.6 报告字段顺序统一

问题：不同机器在 HTML 中同模块字段显示顺序不一致。  
修复：改为按固定字段顺序渲染，提升横向对比体验。

## 6.7 CRLF/LF 兼容增强

问题：在 Linux 运行前经常需要手动 `dos2unix`。  
修复：
- 增加 `source_lib()`：`source` 前自动处理 CRLF
- 增加 `.gitattributes`：约束脚本类文件统一 LF

## 7. 巡检人/脚本编写人信息链路

## 7.1 字段来源

- `SCRIPT_AUTHOR`：固定值 `root_objs`
- `INSPECTOR_NAME`：
  - 优先使用 `--inspector`
  - 未指定时交互输入
  - 非交互环境回退 `whoami`

## 7.2 报告展示

三种报告（html/text/json）都应包含：
- 脚本编写人
- 巡检人

这两项属于审计字段，建议长期保留。

---

## 8. 报告一致性规范（必须遵循）

## 8.1 统一元信息

- report_time
- version
- inspection target count
- script_author
- inspector

## 8.2 统一汇总

- total / ok / warning / critical
- 对应百分比

## 8.3 统一主机维度信息

- 基础身份：`ip` / `hostname` / `status`
- 摘要：`warnings` / `criticals`
- system_info
- core_metrics
- module_status
- raw_output

## 8.4 字段顺序原则

无论采集顺序如何，渲染顺序必须固定，避免“同一报告跨主机展示不一致”。

---

## 9. 远程执行与兼容性设计

## 9.1 远程执行方式

通过 SSH 在目标端运行内嵌脚本并回收输出，避免额外 Agent。

## 9.2 常见兼容点

- `ss` 不存在时回退 `netstat`
- `mpstat` 不存在时走替代方案
- 发行版日志路径差异（`/var/log/secure` vs `/var/log/auth.log`）
- SSH 主机密钥策略兼容

## 10. 运维操作手册

## 10.1 日常巡检建议命令

```bash
# 每日全量巡检（HTML）
./inspect.sh -a -r html --inspector day_shift

# 夜间 JSON 留档
./inspect.sh -a -r json --inspector night_shift

# 问题复核（仅核心模块）
./inspect.sh --host 10.0.0.10 -m cpu,mem,disk,proc --inspector reviewer
```

## 10.2 结果判读建议

- 退出码 `0`：正常
- 退出码 `1`：存在警告，建议 24h 内处理
- 退出码 `2`：存在严重问题，建议立刻介入

## 10.3 报告归档建议

- HTML：面向人工阅读与汇报
- JSON：面向平台接入与二次分析
- Text：面向命令行快速检索与日志系统

## 11. 故障排查手册

## 11.1 连接异常

1. 先手动 SSH 验证
2. 检查账号、端口、防火墙
3. 增大超时重试（`-t 60`）

## 11.2 字段异常或空值

1. 查看 `logs/inspect_*.log`
2. 查看报告内 `raw_output`
3. 检查目标机命令可用性
4. 检查依赖处理选项（是否选择了忽略）

## 11.3 Linux 报 `^M` 类错误

优先排查：
- 是否有人关闭了 `.gitattributes`
- 是否用 CRLF 写入了 `.sh` 文件

# 第三阶段（定时巡检与告警通知）开发总结与运行手册

> 文档目标：把第三阶段“自动化巡检 + 告警通知 + 可观测”能力完整沉淀，便于值班、维护、交接。  
> 适用对象：运维值班、平台维护、后端开发、交付同学。

---

## 1. 阶段目标与能力边界

第三阶段是在前两阶段基础上的“自动化闭环”建设，核心是：

1. **定时巡检自动执行**（Cron/Interval）
2. **异常结果自动告警**（多渠道通知）
3. **巡检与告警可视化**（Grafana 仪表盘）

本阶段不替换第一阶段 Shell 巡检逻辑，而是在 Web 平台侧增加调度、通知、监控闭环能力。

---

## 2. 总体架构（第三阶段视角）

```text
定时任务配置（Web）
  -> APScheduler（后台调度）
  -> InspectorService（调用 inspect.sh / 批量巡检）
  -> Inspection / Alert 落库（PostgreSQL）
  -> NotifyService（钉钉/企微/邮件/Webhook）
  -> Grafana（读取 PostgreSQL 可视化）
```

核心链路说明：
- 调度器触发 -> 批量巡检 -> 识别非 OK -> 推送通知 -> 页面和仪表盘可见。

---

## 3. 关键代码与职责分工

## 3.1 调度服务：`web/services/scheduler_service.py`

职责：
- 初始化 APScheduler
- 新增/删除/暂停/恢复任务
- 启动时恢复历史任务
- 定时任务实际执行（调用 `InspectorService.run_batch_inspection`）

支持任务类型：
- `cron`：5 段表达式
- `interval`：按分钟间隔

关键机制：
- `scheduled_jobs` 配置写入 `settings` 表，保证重启后可恢复
- 使用内存调度器 + DB 配置恢复模式（非 APScheduler JobStore）

## 3.2 通知服务：`web/services/notify_service.py`

职责：
- 读取通知渠道配置
- 统一拼装告警消息
- 分发到不同渠道

支持渠道：
- 钉钉机器人（支持加签）
- 企业微信机器人
- 邮件 SMTP
- 自定义 Webhook

关键机制：
- 通知渠道配置存储在 `settings.key = notify_channels`
- 支持“测试通知”能力，用于配置验收

## 3.3 路由层

- `web/routes/schedule.py`
  - 任务管理 API：新增、列表、删除、暂停、恢复、立即执行
- `web/routes/alerts.py`
  - 告警列表、详情、处理、忽略、批量处理、统计
- `web/routes/settings.py`
  - 通知渠道维护与测试、阈值配置维护

## 4. 定时任务能力详解

## 4.1 任务创建接口行为

`POST /schedule/api/add`

入参关键字段：
- `id`：任务唯一标识（必填）
- `type`：`cron` 或 `interval`
- `cron` 或 `interval`
- `server_ids`：可选，未传则默认全量启用服务器

执行逻辑：
1. 参数校验
2. 生成 Trigger
3. 注册 APScheduler Job（`replace_existing=True`）
4. 持久化任务配置到 `settings.scheduled_jobs`

## 4.2 任务执行逻辑

定时触发时，执行：

1. `InspectorService.run_batch_inspection(server_ids)`
2. 从成功结果里筛选 `status != OK`
3. 调用 `NotifyService.send_inspection_alert(alerts)`

说明：
- 当前失败结果（`failed`）不会直接进入通知内容主干，若需要可扩展“失败告警”通道。

## 4.3 任务生命周期管理

支持操作：
- 列表：`GET /schedule/api/list`
- 暂停：`POST /schedule/api/pause/<job_id>`
- 恢复：`POST /schedule/api/resume/<job_id>`
- 删除：`DELETE /schedule/api/remove/<job_id>`
- 立即执行：`POST /schedule/api/run/<job_id>`

## 5. 通知能力详解

## 5.1 渠道配置存储结构

`notify_channels` 存储为 JSON 数组，每项包括：
- `type`（dingtalk/wecom/email/webhook）
- `name`
- `enabled`
- 渠道特有字段（如 webhook、smtp_host 等）

## 5.2 告警消息格式

通知消息包含：
- 标题（巡检告警）
- 发送时间
- 触发主机列表（最多前 10 条）
- 每台主机关键指标摘要（状态、CPU、内存、磁盘）

## 5.3 各渠道要点

### 钉钉
- 支持签名模式（timestamp + secret HMAC-SHA256）
- 接口响应 `errcode == 0` 视为成功

### 企微
- 通过机器人 webhook 发送 text 消息

### 邮件
- 支持 SMTP + STARTTLS + 登录
- 收件人支持逗号分隔

### Webhook
- 支持 GET/POST
- 可自定义 headers

## 5.4 测试通知

接口：`POST /settings/api/notify/test/<channel_type>`

用途：
- 在正式启用前验证渠道配置
- 便于排查网络和认证问题

## 6. Grafana 可视化能力

## 6.1 预置配置文件

- 数据源：`docker/grafana/provisioning/datasources/datasources.yml`
  - 默认 PostgreSQL `postgres:5432 / inspect`
- 仪表盘加载：`docker/grafana/provisioning/dashboards/dashboards.yml`
  - 自动加载 `/var/lib/grafana/dashboards` 下文件
- 仪表盘定义：`docker/grafana/dashboards/inspect-dashboard.json`

## 6.2 预置仪表盘内容（当前实现）

包含典型监控面板：
- CPU 使用率
- 内存使用率
- 磁盘使用率
- 巡检状态统计
- 最近巡检数据表

说明：
- 数据源均来自 PostgreSQL 的 `inspections`、`servers`、`alerts` 等表。

---

## 7. 数据流与状态流转

## 7.1 巡检结果到告警的流转

1. 定时任务触发巡检
2. 巡检结果落库（`inspections`）
3. 非 OK 项生成/关联告警（`alerts`）
4. 发送通知
5. 告警在 Web 页面处理（HANDLED/IGNORED）

## 7.2 告警状态

- `PENDING`：待处理
- `HANDLED`：已处理
- `IGNORED`：已忽略

## 8. 部署与运行建议（第三阶段）

## 8.1 推荐部署（Docker Compose）

依赖服务：
- Web（Flask + Gunicorn）
- PostgreSQL
- Redis
- MinIO
- Grafana

建议流程：
1. 启动全部容器
2. 先验证 Web 登录与 API
3. 配置通知渠道并发送测试通知
4. 创建一个 interval 任务做小范围验收
5. 观察告警链路与 Grafana 数据

## 8.2 资源建议（起步）

- Web：1C / 512MB
- PostgreSQL：2C / 1GB
- Redis：1C / 256MB
- MinIO：1C / 512MB（按报告容量扩容）
- Grafana：1C / 256MB

## 9. 运维手册（值班视角）

## 9.1 每日巡检检查项

1. `schedule/api/list` 中任务是否有 `next_run_time`
2. 最近巡检是否持续写入 `inspections`
3. `alerts` 待处理数量是否异常增长
4. 通知测试是否可达
5. Grafana 面板是否正常刷新

## 9.2 常见故障处理

### 任务不执行
- 检查 Scheduler 是否启动（Web 启动日志）
- 检查任务是否 pause
- 检查 cron 表达式是否合法
- 检查 server_ids 是否为空或失效

### 通知失败
- 钉钉：检查 webhook、secret、时间戳签名
- 邮件：检查 SMTP 端口、TLS、授权码
- Webhook：检查目标系统可达性与返回码

### Grafana 无数据
- 检查 PostgreSQL 连接
- 检查 SQL 查询时间窗口
- 检查 inspect 任务是否实际写入数据

---

## 10. 安全与合规建议

当前具备：
- 登录鉴权
- 参数清洗
- 任务 API 受登录保护

建议补强：
1. 通知渠道敏感字段（SMTP 密码、机器人密钥）加密存储
2. 操作审计完善（谁新增/暂停/删除了任务）
3. 告警处理增加审批或二次确认（可选）
4. 生产环境限制测试通知频率，避免误告

---

## 11. 验收标准（第三阶段）

建议最小验收集：

1. 创建 Cron 任务，按时触发
2. 创建 Interval 任务，按间隔触发
3. 任务暂停后不再触发，恢复后可触发
4. 立即执行接口可成功返回结果
5. 非 OK 巡检可触发通知
6. 通知测试接口四种渠道至少验证一种成功
7. Grafana 能看到最近巡检数据与状态统计

---

## 12. 当前实现边界与改进方向

## 12.1 当前边界

- 调度任务配置通过 `settings` 记录，不是 APScheduler 原生持久 JobStore
- 批量巡检为同步执行，任务量大时会占用 Web 进程资源
- 通知发送同步执行，渠道慢时会拖长任务执行时间

## 12.2 改进建议（优先级）

1. **高优先级**
   - 调度执行与通知发送异步化（任务队列）
   - 敏感配置加密存储
2. **中优先级**
   - 通知重试与失败队列
   - 告警降噪（聚合、抑制、去重）
3. **低优先级**
   - 值班排班与通知升级策略
   - 异常趋势智能识别

# 报告格式一致性说明（HTML / Text / JSON）

> 目的：明确三种报告格式的字段语义与映射关系，保证展示一致、审计一致、对接一致。

---

## 1. 一致性目标

统一后，`html`、`text`、`json` 三种报告在信息完整性上保持一致：

1. 都有统一的元信息（时间/版本/巡检目标/编写人/巡检人）
2. 都有统一汇总统计（total/ok/warning/critical + 百分比）
3. 都有统一主机维度信息（状态、摘要、系统信息、核心指标、模块状态、原始输出）

---

## 5. 缺省值策略（统一）

为避免不同格式缺字段导致误解，统一缺省值策略：

- 文本类显示：`-`
- JSON 字段：保留字段、值为 `"-"`（字符串场景）
- 摘要字段为空时：
  - `warnings` -> `-`
  - `criticals` -> `-`

---

## 6. 排序一致性策略

HTML 报告已固定字段渲染顺序，Text/JSON 也按同一字段分组输出，避免：
- 同一主机在不同报告格式“看起来不一致”
- 多主机横向比对时字段顺序漂移

---

## 7. 对接建议（平台/数据仓库）

如果要做平台对接或报表汇总，建议优先使用 JSON：

1. 将 `summary` 作为任务维度事实表
2. 将 `servers[]` 展开为主机维度事实表
3. `raw_output` 保留为审计字段，不建议直接做指标统计

---

## 8. 回归测试建议

每次脚本改动后，建议最少执行以下检查：

1. 同一批主机分别输出 `html/text/json`
2. 核对顶部元信息（尤其 `script_author` / `inspector`）
3. 核对每台主机三类报告字段是否齐全
4. 核对汇总统计数值是否一致
5. 核对空值场景的占位符是否一致

---

## 9. 变更要求（新增字段时）

新增任何巡检字段时，必须同步变更：

1. 采集逻辑（模块脚本）
2. 聚合与解析（主脚本）
3. HTML 渲染
4. Text 渲染
5. JSON 输出
6. 本文档映射表

否则视为不完整变更。

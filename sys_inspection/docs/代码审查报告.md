## 2. 本轮发现与处置（按严重等级）

## 2.1 严重问题

### 问题 S1：多进程重复调度，定时任务可能重复执行

- 风险：Gunicorn 多 worker 场景会重复启动 scheduler，导致同一任务重复执行
- 影响：重复巡检、重复告警、重复资源消耗
- 修复：
  - 增加进程锁，确保仅一个进程启动 APScheduler
  - 未抢到锁的进程跳过 scheduler 启动
- 关键位置：
  - `web/services/scheduler_service.py:20`
  - `web/services/scheduler_service.py:33`
  - `web/services/scheduler_service.py:55`

### 问题 S2：巡检服务每次执行重建 Flask app

- 风险：反复 `create_app()` 会重复初始化逻辑、引入副作用与配置漂移
- 影响：稳定性下降，可能触发重复初始化链路
- 修复：
  - `InspectorService` 改为复用当前 app context
  - 无 context 时显式返回错误，不再隐式重建 app
  - 异步线程显式传入 app 对象
- 关键位置：
  - `web/services/inspector_service.py:72`
  - `web/services/inspector_service.py:77`
  - `web/services/inspector_service.py:237`
  - `web/services/inspector_service.py:248`

## 2.2 高问题

### 问题 H1：僵尸进程字段映射不一致

- 风险：远程巡检输出 `PROCESS_ZOMBIE`，Web 入库读取 `ZOMBIE_COUNT`，导致字段丢失
- 影响：`zombie_count` 指标不准，告警判定与展示偏差
- 修复：
  - 入库逻辑优先取 `PROCESS_ZOMBIE`，兼容回退 `ZOMBIE_COUNT`
- 关键位置：
  - `web/services/inspector_service.py:159`

## 2.3 中问题

### 问题 M1：通知渠道 `enabled` 开关未生效

- 风险：配置禁用的渠道仍可能发送通知
- 影响：误通知、骚扰、成本增加
- 修复：
  - 增加 `_channel_enabled` 统一判断
  - 发送前先过滤禁用渠道
- 关键位置：
  - `web/services/notify_service.py:13`
  - `web/services/notify_service.py:43`

### 问题 M2：MinIO 读取失败时 finally 二次异常

- 风险：`response` 未初始化时在 `finally` 调用 close/release，触发二次异常
- 影响：真实错误被覆盖，排障困难
- 修复：
  - `response = None`
  - `finally` 中判空再释放连接
- 关键位置：
  - `web/services/minio_service.py:69`
  - `web/services/minio_service.py:77`

### 问题 M3：非交互场景依赖检测可能卡住

- 风险：Web/定时任务触发 Shell 巡检时进入 `interactive` 读输入分支
- 影响：任务卡死或超时
- 修复：
  - 根据 TTY 自动切换依赖检查模式（有 TTY=interactive，无 TTY=auto）
- 关键位置：
  - `inspect.sh:2015`
  - `inspect.sh:2020`
